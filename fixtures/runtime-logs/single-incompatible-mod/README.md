## Notes

**Acquisition priority: 3rd of 3, deliberately — not because it matters least, but because it is read most safely last.** Capture `clean-base-game` and `single-compatible-mod` first, so this fixture is examined with both a no-mod baseline and a normal-mod baseline already in hand. Reading this log first, with nothing to compare it against, risks misattributing ordinary mod-loading noise — unfamiliar simply because no compatible-mod log has been seen yet — to the actual failure signature. Acquiring it remains the critical-path engineering activity Architectural Resolution must complete before finalizing any signal-recognition design, per `SPRINT10_CAPABILITY_DEFINITION.md`, Section 11's own strengthened precondition — but it should be the third capture, not the first. No plausible-sounding assumption about what a Farming Simulator failure log "probably" contains may substitute for this fixture actually existing.

**Before capturing:** verify the global mods directory contains only the intended mod, for the same reason recorded in `clean-base-game`'s own README — a new savegame does not clear or otherwise affect what the log enumerates from the global mods directory. This was followed: the mods directory was configured to contain only this one mod before capture.

**Mod ZIP characteristics (factual, from direct inspection — the archive itself is not stored in this repository):** declared title "Challenger Hellcat", version `1.0.0.4`, `descVersion="107"`, multiplayer supported, one store item (`type="carFillable"`), no Lua scripts, no declared dependencies. Several notable characteristics suggest the package was distributed without being fully adapted from a source template: its own root vehicle files are named generically (`template.xml`, `template.i3d`, `template.i3d.shapes`) rather than mod-specific; its `<description>` field contains the literal, unedited placeholder text `"Your Mod Name"` in both English and German; and `template.xml` itself retains an authoring comment — `<!-- make sure to match this i3d to your renamed i3d -->` — that was never acted on. None of this is the failure cause recorded above (the declared `descVersion` mismatch is a separate, sufficient, and directly evidenced cause) — it is recorded here as real, observed package characteristics, not as speculation about why the mod failed.

**Resolved:** captured successfully, and the failure signature is unambiguous. The mod's declared `descVersion` (107) exceeds the runtime's own recognized `ModDesc Version` (106) — the same class of mismatch `VersionCompatibilityRule` (Sprint 8) already detects from a mod's own manifest, now observed for the first time in a real runtime log's own rejection message. No dependency issue, no script error, and no asset-loading failure competes with this explanation; the mod was rejected at modDesc validation, before `Load mod:`, before any of its own assets were ever read.
