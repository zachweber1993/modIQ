# Capability Definition

## What is a Version Profile?

A Version Profile is the engineering compatibility context for one supported Farming Simulator release (`VersionProfile.md`). It is not a subsystem and not an aggregate the Assessment owns — `DataModel.md` states this explicitly: *"Unlike the other entities described in this document, a Version Profile is not owned or generated by the Assessment. It is defined and maintained independently of any individual Assessment and is referenced by the Assessment for context."* A Version Profile does not execute logic, perform Assessments, or evaluate Rules (`VersionProfile.md`, Non-Responsibilities) — it *selects and contextualizes* which engineering knowledge, Rules, compatibility patterns, and engine behaviors apply for a given Farming Simulator release, without altering how any of them actually execute.

## What information does it represent?

Per `VersionProfile.md`'s Conceptual Model: a supported game version, applicable engineering knowledge, applicable deterministic rules, supported compatibility patterns, and supported engine behaviors. `modiq-versioning` already names three of these conceptually, as empty stubs: `GameVersion` (supported FS release), `Capability` (supported platform capability), `Compatibility` (version-specific compatibility context), and `VersionProfile` itself (the collection binding them together, per `Glossary.md`'s definition — *"the collection of compatibility information associated with a specific Farming Simulator release. Examples: FS22, FS25, future releases"*).

## What problems does it solve?

Today, modIQ has no mechanism to compare what a mod declares about itself against what the platform actually supports. `XmlCollector` (Sprint 7) can already report *that* a mod declares a `descVersion` — but nothing today reads that value, and nothing today has a notion of "what versions does this platform recognize" to compare it against. A Version Profile capability solves exactly this gap: it gives the platform its first concept of *its own* supported compatibility surface, against which a mod's declared facts can be evaluated.

## What new product capability does it unlock?

For the first time, an Assessment could produce a Finding that directly answers "is this mod compatible with my version of the game?" — the literal Player objective `ProductSpecification.md` names first: **"Compatibility Verification."** No capability shipped through Sprint 7 answers this; every Finding a real Assessment can produce today is either the generic "Evidence was collected" observation or a structural-duplication Warning — neither says anything about version compatibility.

## Which users benefit?

- **Players** — directly, and primarily. "Compatibility verification" is their first-listed objective (`ProductSpecification.md`); this is the single clearest player-facing gap this platform has today.
- **Server Administrators** — secondarily. "Reduce runtime instability" and "improve deployment confidence" both depend on knowing whether a mod collection targets a consistent, supported version.
- **Mod Creators** — secondarily. "Validate compatibility" is a named Creator objective; a Version Profile gives creators the same comparison mechanism players get, applied to their own work before release.

## Which future capabilities depend on it?

- **Version-aware Rule Selection** (`RuleEngine.md`, Execution Principles: *"Version Aware — The Rule Engine evaluates knowledge within the context of the active Version Profile"*; Rule Engine Responsibilities: *"Rule Selection — Determine which Rules are applicable to the current Assessment based on the Assessment Context and Version Profile"*) — named in the frozen specification since Documentation Release 1.0, never implemented, because no Version Profile has ever existed to select against.
- **Version-aware Knowledge Base content** (`KnowledgeModel.md`: *"Knowledge may vary between supported Farming Simulator versions. Version Profiles provide compatibility context without changing the underlying structure of the Knowledge Model"*) — depends on Knowledge Domain integration happening at all, which this capability does not itself provide.
- **Cross-version compatibility analysis** (`Vision.md`, Long-Term Vision) and **multi-version support** (`ProductSpecification.md`, Product Evolution) — both require more than one Version Profile to exist; this capability establishes the first one.

## What deliberately remains out of scope

Stated briefly here; the full list is in Scope Definition, below. Not in scope: a comprehensive multi-version database, exhaustive compatibility knowledge, Knowledge Domain integration of any kind, Rule Selection as a general filtering mechanism, and support for any Farming Simulator release beyond FS25.

## Capability Success Criteria

Per the standing convention this project adopted at Sprint 7 (`PROJECT_HANDOFF_v1.0.md`, Section 6, Principle: *"Sprint 7's own Capability Success Criteria is the standing model for this, expected of every Sprint going forward"*):

**After this Sprint, modIQ can now** compare a mod's own declared target version against the platform's first real, supported compatibility context — for the first time, an Assessment can say something about whether a mod is expected to work with a specific Farming Simulator release, not only that its manifest exists and parses.

Concretely, if Sprint 8 completes as scoped: a user running an Assessment against a mod that declares an unsupported or unrecognized `descVersion` sees a Finding stating so, traceable to both the declared-version Evidence and the active Version Profile that judged it — the direct, evidence-based mechanism behind the Product Specification's explicitly named "Compatibility Verification" objective, unreachable by any capability that exists today.

# Repository Evidence

Reviewed directly against the current tree and frozen specifications, not assumed from any prior session.

## What already exists

- `modiq-versioning`'s three domain files (`capability.rs`, `compatibility.rs`, `game_version.rs`) and `version_profile.rs` are unit structs with a doc comment each and no fields, methods, or logic — confirmed by direct source inspection this session. **`modiq-versioning` has zero dependencies of its own** (`Cargo.toml`: `[dependencies]` is empty) — it does not even depend on `modiq-common` or `modiq-runtime` today, despite `CrateRoadmap.md`'s dependency diagram placing it as a sibling of `modiq-runtime`/`modiq-rules`/`modiq-report` beneath `modiq-common`.
- **Neither `modiq-engine` nor `modiq-rules` depends on `modiq-versioning` today** — confirmed directly against both crates' `Cargo.toml`. Every capability shipped since Sprint 3 (filesystem discovery, archive discovery, XML inspection) extended an *existing* dependency edge (`modiq-collection` → `modiq-runtime`, `modiq-engine` → `modiq-collection`); this capability is the first to require a genuinely **new** cross-crate dependency edge in the workspace graph, in at least one crate and likely two.
- `EngineAPI.md`'s Assessment Service already names *"Version Profiles (once implemented)"* as a declared dependency of the Assessment Service, alongside Knowledge Base — this is not a new architectural surface being invented, but one the frozen specification has been holding a place for since before GOV-004's retirement of the old `EngineAPI` service-object layer. (The retired `VersionProfileService` was an intra-*engine* mediation object, GOV-004's target — its retirement said nothing about whether `modiq-engine` should ever depend on `modiq-versioning` directly; it said only that no intermediate service-object layer should exist. Direct composition remains the approved pattern, per ADR-0010/GOV-004, and applies here exactly as it did to Evidence Collection, the Rule Engine, and Reporting.)
- **`GOVERNANCE.md`'s Crate Boundary Rules section has no entry for `modiq-versioning` or Version Profiles** — verified directly (`grep -n "^## " GOVERNANCE.md` lists Runtime Domain, Rule Engine, Evidence Collection, Reporting, Knowledge Domain, Engine, and CLI; no Version Profile section exists). Every other crate that has ever gained real content (`modiq-collection` at GOV-007, `modiq-rules` implicitly at GOV-004/012) had its boundary rule already recorded before or alongside that content arriving. This is a genuine, previously-unrecorded gap — not a defect in prior work, since `modiq-versioning` has never had a forcing function before now, but a real precondition this capability surfaces.

## A correction to the prior session's own framing

`SPRINT8_INITIALIZATION_REPORT.md` and `REPOSITORY_CLOSEOUT_REPORT.md` (Sprint 7) both describe Sprint 7 as producing "declared FS version" data alongside declared dependencies. **Direct inspection of `crates/modiq-collection/src/collection/xml_collector.rs` this session shows this is not accurate as implemented.** `XmlCollector::inspect` extracts exactly one kind of fact from a well-formed manifest: `<dependency>` element text content. It does not read the `descVersion` attribute at all — even though every test fixture in the same file includes `descVersion="93"` on the root `<modDesc>` element, that attribute is never consulted by any production code. **Declared dependency data is real, extracted, tested Evidence. Declared FS version data does not yet exist anywhere in this platform's Evidence.** This is a material correction: Version Profile-aware compatibility checking cannot compare a mod's declared version against anything until a Collector actually extracts one — this capability's first concrete implementation dependency is smaller in scope than "read `modDesc.xml`" (already done) and larger than "the version data already exists" (it does not).

## Architectural tensions the frozen specification itself contains

Two textual tensions in already-Frozen documents bear directly on how this capability should integrate — neither is a defect to silently resolve; both are flagged for Chief Architect resolution in Architectural Evaluation, below, per this project's own discipline of recording rather than quietly reconciling a specification ambiguity.

1. **`DataModel.md`'s Entity Relationships diagram** lists `Assessment`'s relationships as a flat set of siblings: *"has one Assessment Subject / has one Assessment Context / ... / uses one Version Profile"* — Version Profile as a direct relationship of `Assessment`, not nested under Context. But the same document's own **Assessment Context** entity description lists *"Version Profile"* as merely one illustrative example of what Context "may include," alongside timestamp, configuration, platform capabilities, and execution environment. The diagram and the prose description do not obviously agree on where Version Profile actually attaches.
2. **`RuleEngine.md`'s Rule Selection responsibility** names Version Profile as an input alongside Assessment Context, not subsumed by it (*"based on the Assessment Context and Version Profile"* — two named inputs, not one), reinforcing the diagram's reading over the Context prose's. Yet the same document's own **Relationship to the Runtime Domain** section, listing exactly which runtime entities the Rule Engine consumes (*"Assessment, Assessment Context, Assessment Subject, Evidence"*), omits Version Profile entirely — despite the Version Aware execution principle two sections earlier requiring the Rule Engine to evaluate knowledge "within the context of the active Version Profile."

## Current `Assessment` shape (verified against source)

`Assessment::new(subject: AssessmentSubject, context: AssessmentContext)` — both parameters are still content-free unit structs, exactly as they have been since before Sprint 1; `AssessmentContext` in particular has never carried real content, unlike Evidence/Finding/Recommendation, which gained real fields at Sprint 2. Every call site across `modiq-runtime`, `modiq-engine`, `modiq-cli`, and the Sandbox constructs `AssessmentContext` as the bare unit value. Any design that gives Version Profile a real presence on `Assessment` requires `AssessmentContext` (or `Assessment` itself) to gain real content for the first time in the platform's history — a change with a footprint at least as large as Sprint 2's own Evidence/Finding/Recommendation content work, not a small addition.

## Current `RuleEngine::evaluate` shape (verified against source)

`RuleEngine::evaluate(&self, evidence: &[Evidence]) -> Vec<RuleOutcome>` — no Rule Selection step exists today in any form. Both real Rules (`EvidencePresenceRule`, `StructuralDuplicationRule`) are dispatched unconditionally, every time, per GOV-012's own resolution (*"Rules compose independently; no suppression model exists"*). "Rule Selection" as `RuleEngine.md` describes it — determining *which* Rules are applicable based on Context and Version Profile — has never been built; today, every Rule is always applicable.

# Architectural Evaluation

**Protecting the Evidence philosophy is straightforward here, and requires no new judgment call** — `EvidenceCollection.md`'s own Non-Responsibilities already states the answer directly: *"Interpreting game-version-specific behavior on its own. Where interpretation depends on a Farming Simulator version, that context is supplied via Assessment Context / a future Version Profile... not decided internally by Evidence Collection."* Collectors extract facts; they do not judge compatibility. Concretely, this means the only Evidence Collection change this capability needs is additive and purely factual: **extend `XmlCollector` to also extract the `descVersion` attribute as its own `XmlInspection` Evidence item**, following the identical Content Extraction discipline Sprint 7 Phase 3 already established for `<dependency>` elements. `XmlCollector` reports *that* a mod declares `descVersion="93"`; it does not, and must not, decide whether `93` is supported. That judgment belongs later in the pipeline, exactly as the mission's own framing requires.

## Fork 1 — Where does the Version Profile attach to `Assessment`?

Two options, both textually defensible against the tension identified in Repository Evidence above:

**Option A — Direct `Assessment` field**, matching the Entity Relationships diagram literally: `Assessment::new` gains a third parameter (`version_profile: VersionProfile` or similar), sibling to `subject` and `context`. `Assessment::version_profile()` becomes a new read-only accessor, following the exact precedent `subject()`/`context()` already set.

**Option B — Carried inside `AssessmentContext`**, matching Context's own descriptive prose (*"Assessment Context describes the environment... may include: Version Profile..."*): `Assessment::new`'s own signature is unchanged; `AssessmentContext` gains its first real field, and `Assessment::context().version_profile()` exposes it.

**Recommendation:** Option A. The Entity Relationships diagram is the more structurally precise statement (a dedicated bullet, not folded into a list of Context examples), and `RuleEngine.md`'s Rule Selection responsibility naming Version Profile as a second, independent input alongside Assessment Context — not a sub-property of it — is corroborating evidence toward the same reading, not merely a coincidence of two documents drafted separately. Recommended, not decided: this is precisely the kind of specification-reading judgment call this project's history assigns to Chief Architect review (`CHIEF_ARCHITECT_HANDOFF_v1.0.md`, Section 3: "Domain model integrity"), not to Lead Engineer implementation discretion — see Chief Architect Questions, below.

Either option requires `AssessmentContext` to remain content-free (Option A) or finally gain content (Option B) — but both require every existing call site constructing `Assessment::new` to change, since `AssessmentContext`'s own unit-struct shape is untouched only under Option A. This is a Level 3/4 change under either reading (`GOVERNANCE.md`'s Change Categories: *"aggregate redesign"* territory) and, unlike every capability since Sprint 3, cannot be made purely additively — `Assessment::new`'s existing two-parameter signature does not survive Option A unchanged, and every one of its current callers (`modiq-runtime`'s own tests, `modiq-engine`, `modiq-cli`, the Sandbox) needs updating regardless of which option is chosen.

## Fork 2 — Where does version-aware evaluation actually happen?

The mission's own framing — *"Version-aware evaluation should occur later in the assessment pipeline"* — is well-supported by repository evidence and not in tension with anything found this session. The open question is *how much* Rule Engine machinery that requires:

**Option A — Build real Rule Selection**, matching `RuleEngine.md`'s literal responsibility description: `RuleEngine::evaluate` gains a filtering step that excludes Rules based on Version Profile/Context before dispatch. This is the specification's own named responsibility, fully realized for the first time.

**Option B — A single new Rule that itself consults the Version Profile as an evaluation input**, alongside Evidence, following `StructuralDuplicationRule`'s exact shape otherwise: no Rules are excluded or filtered; the new Rule simply receives the active Version Profile (however Fork 1 resolves) and compares it against declared-version Evidence, producing a Finding only when relevant Evidence exists. Every existing Rule is unaffected.

**Recommendation:** Option B, on the same "capability before abstraction" grounds this project has applied at least seven times before (`PROJECT_HANDOFF_v1.0.md`, Section 6). Three Rules total, after this capability, is not evidence a general selection/filtering mechanism is needed — no Rule has ever needed to be excluded from evaluation; the only reason to build Option A now would be that `RuleEngine.md` names it, which this project's own history (GOV-004, GOV-012, GOV-013) has repeatedly held is not sufficient justification on its own. Option B requires `RuleEngine::evaluate`'s signature to gain a new parameter (the active Version Profile, or an `Option` of one) — itself a public-API and Rule-dispatch-shape decision in the same family GOV-012 already resolved once, and arguably warranting its own Governance Register item before implementation, mirroring GOV-012's own precedent (design questions resolved by governance, then implemented) — not opened here, per this session's explicit scope.

## Evidence Collection

Unaffected in boundary, extended in content only (`descVersion` extraction, above). No Collector decides compatibility; every Collector remains purely observational, satisfying the mission's own instruction without requiring any exception or justification.

## Rule Engine

See Fork 2. Whichever option is chosen, the existing Evidence-Based and Explainable execution principles apply unchanged: a version-compatibility Finding must trace to both the declared-version Evidence and the Version Profile that judged it, and must never assign `Error` severity for anything less than Evidence conclusively establishes (`DataModel.md`: Finding Severity). An unsupported declared version is very plausibly a `Warning`, not an `Error` — this platform cannot know, from XML content alone, whether an unrecognized `descVersion` actually breaks the mod at runtime, only that it does not match a Version Profile the platform recognizes.

## Knowledge Model

No interaction, and none should be introduced. `modiq-knowledge` remains zero-implementation scaffolding with zero forcing function; this capability does not create one. `KnowledgeModel.md`'s own "Version Aware" principle describes a future where Knowledge varies by version — that future is not this Sprint's concern, and building toward it now would repeat exactly the speculative-scaffolding failure mode `CHIEF_ARCHITECT_HANDOFF_v1.0.md` Section 8 names.

## Assessment Report

No shape change. A version-compatibility Finding (however produced) flows through `AssessmentReport`'s existing `evidence()`/`findings()`/`recommendations()` lists, consistent with this project's "Evidence-first, single-channel reporting" principle — no second reporting channel for version-compatibility facts.

## Existing architectural boundaries

- **A new Crate Boundary Rule for `modiq-versioning` is a genuine prerequisite**, not previously needed. Recommended shape, for Chief Architect consideration rather than adoption here: Version Profiles *own* `GameVersion`, `Capability`, `Compatibility`, and `VersionProfile` definitions; they must never execute Rules, perform Assessments, mutate Runtime state, or produce Findings/Recommendations directly (`VersionProfile.md`'s own Non-Responsibilities already state this in full; the gap is that `GOVERNANCE.md` has never carried the equivalent enforceable boundary statement).
- **New dependency edges.** `modiq-engine` would depend on `modiq-versioning` for the first time (to compose it into `AssessmentService`, matching `EngineAPI.md`'s already-declared but never-implemented dependency). `modiq-rules` would depend on `modiq-versioning` for the first time only if Fork 2 resolves toward Option B (the new Rule needs the `VersionProfile` type to accept it as an evaluation input). Both are new edges in a dependency graph that has not changed shape since Sprint 3 (`modiq-collection`'s introduction) — worth naming explicitly as a bigger structural change than any capability shipped since.
- **`AssessmentService`'s public entry points.** Under Fork 1 Option A, `Assessment::new` changes — `execute` and `execute_from_assessment_input` both construct an `Assessment` internally and would need a new parameter of their own to pass a `VersionProfile` through, an unavoidable, visible signature change to both of `AssessmentService`'s existing entry points. This is a materially different situation from every prior capability, each of which (per GOV-008's own three-Sprint-running deferral) added capability *without* touching either entry point's signature. Whether this finally forces GOV-008's resolution, or whether a third, additive entry point is introduced instead (mirroring how `execute_from_assessment_input` was itself added alongside `execute` rather than replacing it), is squarely a Chief Architect Review Question, not resolved here.

# Implementation Strategy

## Minimum viable Version Profile

A single, hardcoded FS25 `VersionProfile` — one `GameVersion` value, one small set of `descVersion` values the platform recognizes as supported — constructed by the application layer (Sandbox, CLI) exactly the way `AssessmentSubject`/`AssessmentContext` are constructed by callers today, no persistence, no configuration file, no profile-selection UI, and no second profile. This mirrors Sprint 7's own "smallest real slice" discipline (`SPRINT7_CAPABILITY_AND_IMPLEMENTATION_PLAN.md`'s Phase 2, "the minimal, real, testable slice") applied to a domain, not a Collector.

## Phased approach

Narrow enough that each phase can be separately authorized, mirroring every prior Sprint's own plan shape — not a single blanket approval.

**Phase 1 — Design Preparation and Governance.** Resolve Fork 1 (Assessment field vs. Context field) and Fork 2 (Rule Selection vs. single version-aware Rule) by Chief Architect decision; very likely produces a new Governance Register item (a "Version Profile Integration Model," analogous in shape to GOV-009/GOV-011/GOV-012) recording both resolutions, plus the new `modiq-versioning` Crate Boundary Rule. No code this phase.

**Phase 2 — `modiq-versioning` minimum domain content.** Give `GameVersion` and `VersionProfile` real fields for the first time since Sprint 0 — e.g., `GameVersion` wrapping a version identifier, `VersionProfile` holding a `GameVersion` plus a small, explicit set of supported `descVersion` values, with a query method (`supports(descVersion) -> bool` or similar). `Capability`/`Compatibility` are not required for the minimum viable slice and can remain deferred unless Phase 1's resolution needs them. Whether `ADR-0007`'s Runtime Entity Design Pattern (identity, opaque references, constructor validation) transfers to a Knowledge/Version-domain type rather than a Runtime aggregate member is worth explicit Chief Architect confirmation — `VersionProfile` is not owned by `Assessment` the way Evidence/Finding/Recommendation are, so not every element of that pattern obviously applies.

**Phase 3 — `XmlCollector` extension: declared version extraction.** Add `descVersion` extraction as a new, purely additive `XmlInspection` Evidence item, following Sprint 7 Phase 3's exact discipline (a new factual observation, not an interpretation). Independently testable and mergeable ahead of any Rule or Engine wiring, exactly as Sprint 7's own Phase 2/3 split demonstrated.

**Phase 4 — Wiring.** `modiq-engine` gains its first dependency on `modiq-versioning`; `Assessment` (or `AssessmentContext`, per Phase 1's resolution) carries the `VersionProfile` reference; `AssessmentService`'s entry points plumb it through, per Phase 1's resolution of whether that's a breaking or additive change.

**Phase 5 — Rule.** Implement the version-compatibility Rule (if Fork 2 resolved toward Option B) following `StructuralDuplicationRule`'s exact shape — `modiq-rules` gains its first dependency on `modiq-versioning`.

**Phase 6 — Closeout.** Full-workspace validation, documentation sync, and — explicitly, correcting the two-Sprint-running pattern Engineering Release 0.7 itself named as a risk not to repeat a third time — an Engineering Release record produced at or near this Sprint's own Closeout, not after it.

## Reusable infrastructure

`StructuralDuplicationRule`'s shape as the direct template for the new Rule; Sprint 7's Content Extraction discipline (Phase 3) as the direct template for `descVersion` extraction; the additive-entry-point precedent (`execute` / `execute_from_assessment_input`) as a candidate escape hatch if Fork 1/Engine wiring should avoid a breaking change to existing callers.

# Scope Definition

## In Scope

- The `VersionProfile` conceptual model gaining real, minimal content in `modiq-versioning` (a single FS25 profile, a small supported-`descVersion` set).
- `XmlCollector` extended to extract a mod's declared `descVersion` as new, factual `XmlInspection` Evidence.
- Resolving Fork 1 (Assessment/Context attachment) and Fork 2 (Rule Selection vs. single Rule) via Chief Architect decision and, likely, a new Governance Register item.
- `AssessmentService` composing `modiq-versioning` for the first time, by direct composition (ADR-0010/GOV-004's established pattern) — no intra-engine service object.
- Exactly one new Rule, comparing a mod's declared version against the active Version Profile — if Fork 2 resolves toward Option B.
- A new `modiq-versioning` Crate Boundary Rule in `GOVERNANCE.md`.

## Out of Scope

- A comprehensive, multi-version compatibility database — one FS25 profile only.
- Exhaustive compatibility knowledge (engine behaviors, compatibility patterns, best practices) — `Capability`/`Compatibility` remain deferrable unless Phase 1 finds otherwise.
- General-purpose Rule Selection as a filtering mechanism (Fork 2 Option A) — deferred pending real evidence of need beyond this one capability, per this project's own "capability before abstraction" standard.
- Knowledge Base population or Knowledge Domain integration of any kind.
- Any Farming Simulator release other than FS25.
- Any profile-selection UI, configuration file, or persistence mechanism — the single default profile is supplied by the application layer exactly as `AssessmentSubject`/`AssessmentContext` are today.
- Resolving GOV-008 as a side effect — if the Engine wiring forces a breaking `AssessmentService` signature change, that should be its own explicit governance decision, not an incidental consequence of this capability.

# Risks

- **Two genuine architectural forks (Fork 1, Fork 2) are load-bearing, not implementation details.** Building Phase 4 (Wiring) or Phase 5 (Rule) ahead of Phase 1's explicit resolution risks the exact "implementation-led architecture" failure mode this project's Decision Framework exists to prevent (`CHIEF_ARCHITECT_HANDOFF_v1.0.md`, Section 6, Section 8).
- **This capability cannot avoid a breaking change somewhere.** Unlike every capability shipped since Sprint 3, `AssessmentContext`'s own content-free shape (or `Assessment::new`'s signature, under Fork 1 Option A) does not survive this work unchanged. GOV-008's three-Sprint deferral has so far been sustained specifically by capabilities that never needed to touch it; this is the first candidate that plausibly does, and that tension should be surfaced explicitly during Phase 1, not discovered mid-implementation.
- **The prior session's "declared FS version" framing was inaccurate**, as this session's own direct source inspection found (Repository Evidence, above). Any Sprint 8 authorization based on the assumption that version data already exists in Evidence would be authorizing against a false premise; Phase 3 (`descVersion` extraction) is a real, necessary prerequisite, not a formality.
- **New cross-crate dependency edges** (`modiq-engine` → `modiq-versioning`, possibly `modiq-rules` → `modiq-versioning`) are the first structural change to the dependency graph since Sprint 3's introduction of `modiq-collection` — worth deliberate review, not assumed safe by analogy to adding an external dependency (as Sprint 7's `roxmltree` addition was).
- **Scope drift toward exhaustive version support or Knowledge Domain integration** is a realistic failure mode given how much of `Vision.md`'s long-term direction this capability touches — this plan's explicit Scope Definition exists specifically to make that drift visible early, the same purpose Sprint 7's own out-of-scope list served.
- **Severity discipline for the new Finding.** An unsupported declared version is Evidence of a mismatch, not conclusive proof of breakage — assigning `Error` rather than `Warning` would violate `DataModel.md`'s Evidence-Based severity discipline (*"a Rule must never assign a severity stronger than what its Evidence conclusively establishes"*) and should be checked explicitly during Implementation Review, not assumed correct because it "feels" like an error.
