# 1. Repository Evidence

Gathered this session by direct inspection — cited precisely, not carried forward from `SPRINT8_CAPABILITY_AND_IMPLEMENTATION_PLAN.md`'s own summary of it, since several of this session's conclusions rest on evidence that plan only partially surfaced.

## Ownership precedent (ADR-0003)

`ADR-0003` (Assessment as the Aggregate Root) enumerates Assessment-owned entities explicitly: *"Examples of Assessment-owned entities include: Assessment Context, Assessment Subject, Evidence, Findings, Recommendations, Assessment Report."* **Version Profile is absent from this list.** This is independent, corroborating evidence for `DataModel.md`'s own explicit ownership exception (*"Unlike the other entities described in this document, a Version Profile is not owned or generated by the Assessment... it is referenced by the Assessment for context"*) — two separately-authored Documentation Release 1.0 documents, both dated 2026-07-16, converge unprompted on the same reading. This project's own governance discipline treats unprompted convergence across independent sources as materially stronger evidence than either source alone (`GOVERNANCE.md`, GOV-004's own resolution rested on exactly this kind of convergence).

## Entity design precedent (ADR-0007)

`ADR-0007` (Runtime Entity Design Pattern) is explicitly scoped to entities `Assessment` **owns** (*"Evidence, Finding, and Recommendation are owned exclusively by Assessment"*) — its Entity Identity, Opaque Runtime References, and Aggregate-Owned Relationship Resolution provisions all presuppose ownership. Since Version Profile is, by ADR-0003's own list and `DataModel.md`'s explicit statement, **not** owned by `Assessment`, ADR-0007's pattern does not straightforwardly govern how `Assessment` should reference it — a distinct design question from how `modiq-versioning`'s own internal types should be shaped.

## API-evolution precedent (ADR-0009 / GOV-008)

`ADR-0009` records, without deciding, that `AssessmentService`'s public entry point would eventually require a breaking signature change once Evidence Collection was implemented — explicitly *"does not authorize implementing it now, and does not select among the ways such a change could be shaped."* GOV-008 has tracked that unresolved design question ever since, deliberately Open across five Sprints, resolved only by the entry point remaining unchanged and a second, additive entry point being introduced instead. This is the closest direct precedent in this repository for a decision of exactly Decision 4's shape, below — a foreseeable, necessary breaking change, recorded before implementation, with its exact shape deliberately left open pending governance approval.

## `Assessment::new`'s stability record (verified against source)

`Assessment::new(subject: AssessmentSubject, context: AssessmentContext)` has held this exact two-parameter shape since before Sprint 1. Sprint 2 — the only prior Sprint to give any Assessment-adjacent type real content — changed `Evidence`, `Finding`, and `Recommendation`'s own constructors, but never touched `Assessment::new`'s signature, because those three are added to the Assessment *after* construction (`add_evidence`, `add_finding`, `add_recommendation`), not supplied as constructor parameters. `AssessmentSubject` and `AssessmentContext`, by contrast, **are** constructor parameters, and neither has changed shape even once across seven Sprints. Any change here breaks a streak with no precedent in this codebase.

## Collector Composition precedent (Sprint 7)

`COLLECTOR_COMPOSITION_ARCHITECTURE_PROPOSAL.md`'s Chief Architect Decision Record set a concrete, five-condition threshold for when inline Collector composition should be reconsidered, including *"three or more content Collectors typically participate in an Assessment."* Relevant to Decision 2, below: introducing a second Collector to extract one additional attribute from a document `XmlCollector` has already opened and parsed would move toward, without reaching, that threshold for no corresponding capability gain.

## Dependency graph (verified against `Cargo.toml`, all four relevant crates)

- `modiq-versioning/Cargo.toml`: `[dependencies]` is empty. Zero dependencies, including no dependency on `modiq-common`, despite `CrateRoadmap.md`'s dependency diagram drawing that edge.
- `modiq-engine/Cargo.toml`: depends on `modiq-runtime`, `modiq-rules`, `modiq-report`, `modiq-collection`. No dependency on `modiq-versioning`.
- `modiq-rules/Cargo.toml`: depends only on `modiq-runtime`.
- **No crate in the workspace depends on `modiq-versioning`, and `modiq-versioning` depends on nothing.** It is the only workspace crate with zero edges in either direction.

## `modiq-common`'s actual content (verified against source)

`crates/modiq-common/src/id.rs` and `error.rs` each contain a single doc comment (*"Shared identifier types."* / *"Shared error types."*) and no code. `AssessmentId`'s actual identity mechanism (a process-local `AtomicU64` counter) is implemented directly inside `modiq-runtime`, not shared through `modiq-common` at all. A repository-wide `grep` for `modiq_common`/`modiq-common` finds it referenced only in its own `Cargo.toml` — **no other crate depends on it, references it, or has ever needed it.** GOV-003 ("Role of `modiq-common`") remains Open for exactly this reason, unchanged since Engineering Release v0.1.0-alpha.

## Governance boundary gap (verified against `GOVERNANCE.md`)

`grep -n "^## " GOVERNANCE.md`'s Crate Boundary Rules confirm sections exist for Runtime Domain, Rule Engine, Evidence Collection, Reporting, Knowledge Domain, Engine, and CLI. **No section exists for Version Profiles / `modiq-versioning`.**

# 2. Architectural Decision 1 — Version Profile Ownership

**Restated question:** Should `VersionProfile` be (A) a first-class property of `Assessment`, (B) part of `AssessmentContext`, or (C) another architecture?

## Alternatives

**A — Direct `Assessment` field.** `Assessment::new` gains a third parameter, sibling to `subject` and `context`; a new `version_profile()` accessor follows the exact precedent `subject()`/`context()` already set.

**B — Nested inside `AssessmentContext`.** `Assessment::new`'s own signature is unchanged; `AssessmentContext` gains its first real field, exposed via `Assessment::context().version_profile()`.

**C — Not stored on `Assessment` at all; supplied only as an orchestration-time parameter.** `AssessmentService` accepts a `VersionProfile` and passes it directly to whichever pipeline stage needs it (per Decision 3), without `Assessment` itself ever holding a reference.

## Evaluation

**Domain semantics.** `VersionProfile.md`'s own Conceptual Model states a Version Profile *"selects and contextualizes"* engineering knowledge and Rules — it describes a relationship *of the Assessment*, not a descriptive attribute *of the Assessment's environment* the way Context's other examples (timestamp, configuration, platform capabilities) are. `DataModel.md`'s Assessment Context entity description lists Version Profile as one example among several environmental facts — but this is the looser of two statements in tension (see Repository Evidence, Sprint 8 Capability Plan). `DataModel.md`'s Entity Relationships diagram, and `RuleEngine.md`'s Rule Selection responsibility (naming Version Profile as a second, independent input *alongside* Assessment Context, not folded into it), both treat it as structurally distinct.

**Lifecycle.** Per `VersionProfile.md`: *"The platform architecture is independent of any specific Farming Simulator release... a Version Profile provides compatibility context."* Per `DataModel.md`: *"The Version Profile exists independently and is only referenced by the Assessment."* Its lifecycle spans many Assessments, unlike Context and Subject, which (per `ADR-0007`'s Value Objects principle, applied to Runtime entities generally) are plain, non-identity-bearing values scoped to the single Assessment that constructs them.

**Ownership.** `ADR-0003`'s own enumerated list of Assessment-owned entities omits Version Profile — a second, independent piece of evidence (see Repository Evidence, above) corroborating `DataModel.md`'s explicit exception clause. Two Documentation Release 1.0 documents converging unprompted on the same distinction is the same class of evidence GOV-004's resolution treated as decisive.

**Identity.** `VersionProfile.md`'s Traceability section requires: *"Every Assessment should record the Version Profile under which it was executed"* and *"Historical Assessment results must remain reproducible using the Version Profile active when the Assessment was executed."* This is an identity/reference requirement, not a descriptive one — closer in kind to how `Finding` must carry a `RuleReference` than to how Context might carry a timestamp. If Version Profile itself gains a stable identity in the future (a `VersionProfileId`, mirroring `AssessmentId`), a buried Context field is a weaker foundation for that than a named, first-class relationship.

**Architectural consistency.** Option C would satisfy the mission's own instruction that "version-aware evaluation should occur later in the pipeline" for the *evaluation* itself, but it directly conflicts with `VersionProfile.md`'s Traceability requirement above: if `Assessment` never records which Version Profile was active, a completed Assessment cannot later answer "what was this judged against" — the same reproducibility guarantee `DataModel.md`'s Immutability principle already requires of every other Assessment-adjacent fact. Option C is evaluated and **not recommended**, for this reason specifically, not because orchestration-level use is wrong (Decision 3 still routes the actual comparison through the Rule Engine; Option C's flaw is *only* that it does not additionally record the reference on `Assessment` itself).

**Future extensibility.** Option A generalizes cleanly if the platform ever needs `Assessment` to reference another externally-maintained-but-not-owned resource — each becomes its own named sibling relationship, mirroring this one, rather than accumulating inside an increasingly undifferentiated Context. Option B risks Context becoming an informal catch-all, the kind of undifferentiated growth this project's crate- and entity-boundary discipline has otherwise avoided everywhere else.

## Recommendation

**Option A** — a direct, first-class `Assessment` field, structurally sibling to `subject` and `context`, not nested inside either. Grounded in convergent evidence (`ADR-0003`'s ownership list, `DataModel.md`'s ownership exception, `RuleEngine.md`'s Rule Selection wording all independently pointing the same direction) rather than a single reading of one ambiguous passage.

## Implementation consequences

`Assessment::new` gains a third constructor parameter; every existing call site (`modiq-runtime`'s own tests, `modiq-engine`, `modiq-cli`, the Sandbox) requires updating. `AssessmentContext` remains content-free under this option — its own first-content moment is deferred, not forced by this capability. `ADR-0003` itself would need a cross-reference amendment recording the new relationship, following this project's established amendment discipline (`Architecture.md`, `EngineAPI.md`, `EvidenceCollection.md`, `DataModel.md` have each been amended this way already).

# 3. Architectural Decision 2 — Version Extraction

**Restated question:** Where should `descVersion` extraction occur?

## Alternatives

**A — Extend `XmlCollector`.** Add a `descVersion`-reading step inside the same `inspect()` method that already parses the manifest into a `roxmltree::Document` and already extracts `<dependency>` elements from it.

**B — A new, dedicated Collector.** A second Collector solely responsible for version extraction, composed alongside `XmlCollector` per Sprint 7's already-approved multi-Collector composition model.

**C — `AssessmentService` (orchestration) reads the manifest directly**, outside any Collector.

**D — The Rule Engine reads raw manifest content directly** at evaluation time, bypassing Evidence Collection.

## Evaluation against the Evidence philosophy ("Collectors observe, Rules evaluate")

**Option A.** `XmlCollector` has already opened, read, and validated the manifest as well-formed XML by the time `inspect()` runs; reading one additional attribute of the same, already-parsed `roxmltree::Document` requires no new file location, no new parsing, no new dependency. This is a purely factual, deterministic observation — extracting *that* a mod declares `descVersion="93"` is architecturally identical in kind to extracting *that* it declares a dependency, which this platform already treats as Evidence Collection's job. Satisfies the Collector Contract's Determinism, Factual Accuracy, and Boundary Respect guarantees exactly as the existing dependency extraction does, and requires no revisiting of Sprint 7's already-approved Collector Composition architecture.

**Option B.** Would duplicate `XmlCollector`'s existing file-location and parsing logic in a second Collector, solely to read one more attribute of the same document a sibling Collector has already opened — no concrete forcing function justifies this duplication, and it edges the platform toward (without individually reaching) the Collector Composition Architecture Proposal's own named extraction threshold (*"three or more content Collectors"*) for no corresponding capability gain. Weighed and not recommended.

**Option C.** Directly violates `EngineAPI.md`'s own stated boundary — *"Performs no content inspection itself — Evidence production belongs exclusively to Evidence Collection"* — and `GOVERNANCE.md`'s Engine boundary rule. This is the same class of violation the Sprint 7 Collector Composition study rejected Option B (inter-Collector dependency) for: a boundary already frozen, not a judgment call available to this Sprint.

**Option D.** Directly violates `RuleEngine.md`'s own frozen boundary — *"Evidence is produced by the Evidence Collection subsystem... not by the Rule Engine... The Rule Engine consumes Evidence, it never produces it"* — and is the precise failure mode this session's own instruction ("Collectors observe, Rules evaluate") exists to prevent. Rejected outright.

## Recommendation

**Option A.** The dominant option on every axis evaluated — smallest change, zero new architectural surface, zero re-litigation of an already-approved composition model, and the direct, additive continuation of Sprint 7 Phase 3's own Content Extraction precedent.

## Implementation consequences

A new `XmlInspection` Evidence variant-of-fact (found-and-recognized, found-but-unrecognized, or absent `descVersion`) is added to `XmlCollector::inspect`, following the same found/malformed/absent shape already used for the manifest's own well-formedness. No new external dependency, no new Collector, no change to `AssessmentService`'s Collector composition.

# 4. Architectural Decision 3 — Version-aware Evaluation

**Restated question:** Where should Version Profile first influence an Assessment's outcome?

## Alternatives

**A — `AssessmentService` (orchestration).** The Engine directly compares declared-version Evidence against the active `VersionProfile` and constructs a Finding itself.

**B — Rule Engine.** A Rule — existing or new — evaluates declared-version Evidence using the active `VersionProfile` as an additional input, producing a `RuleOutcome` through the established mechanism.

**C — Knowledge Model.** Compatibility logic is represented as a new Knowledge Domain entity type, consulted by the Rule Engine.

**D — A dedicated Compatibility Service**, a new subsystem distinct from the Rule Engine.

## Evaluation, preserving Evidence / Interpretation / Assessment / Reporting separation

**Option A.** Violates `GOVERNANCE.md`'s Engine boundary (*"Must never: implement business rules"*) and `RuleEngine.md`'s own ownership of Finding Generation and Evidence Evaluation. Concretely: `Finding::new` requires a `RuleReference` (verified against `modiq-runtime`'s constructor and test code) — a Finding produced outside the Rule Engine has no real Rule to reference, breaking Traceability Management (`RuleEngine.md`: *"maintain conceptual relationships between... Rules, Evidence, Findings, Recommendations"*) at its root. Rejected.

**Option B.** The textually mandated location: `RuleEngine.md`'s own Version Aware execution principle (*"The Rule Engine evaluates knowledge within the context of the active Version Profile"*) and Rule Selection responsibility name exactly this. Preserves every existing boundary unchanged — Evidence Collection still only observes (Decision 2 supplies the fact); the Rule Engine still, and only, evaluates; Reporting still only reflects. This is the boundary the mission's own instruction ("maintain existing separation") already implies, made concrete.

**Option C.** `KnowledgeModel.md`'s own text — *"Version Profiles provide compatibility context without changing the underlying structure of the Knowledge Model"* — describes Version Profile as contextualizing Knowledge, not becoming Knowledge itself. Adopting this option now would force `modiq-knowledge` into its first real implementation with zero forcing function beyond this single capability, repeating the exact "speculative extensibility" failure mode this project has already named and avoided (`CHIEF_ARCHITECT_HANDOFF_v1.0.md`, Section 8). Not rejected as conceptually wrong forever — `KnowledgeModel.md` explicitly anticipates version-varying Knowledge eventually — but rejected as this Sprint's minimum-viable mechanism.

**Option D.** Reintroduces exactly the kind of intra-engine service-object layer GOV-004/ADR-0010 evaluated and retired (the original `VersionProfileService` scaffolding). Building a dedicated service now, for one comparison, in support of one Rule, is the same premature-abstraction pattern this project's own "capability before abstraction" discipline has rejected at least seven times already. Rejected.

## Recommendation

**Option B** — the Rule Engine, specifically via a Rule that receives the active `VersionProfile` as an evaluation input alongside Evidence (the narrower of the two sub-variants `SPRINT8_CAPABILITY_AND_IMPLEMENTATION_PLAN.md` already identified as Fork 2), rather than building general-purpose Rule Selection filtering. Three Rules total after this capability is not, on its own, evidence a filtering mechanism is needed — no Rule has ever required exclusion from evaluation.

## Implementation consequences

`RuleEngine::evaluate`'s signature gains a new input (the active `VersionProfile`, or an `Option` of one) — itself a change in the same family GOV-012 already resolved once for Rule dispatch shape, and one this session recommends treating with the same governance rigor (see Architectural Decision 6). A new Rule is added, following `StructuralDuplicationRule`'s exact shape. `modiq-rules` gains its first dependency beyond `modiq-runtime`.

# 5. Architectural Decision 4 — Assessment Construction

**Restated question:** Should `Assessment` evolve its constructor? Should another construction mechanism exist? Should initialization responsibility move elsewhere? Is this normal domain evolution, or an architectural boundary change?

## Evaluation

**Is this normal domain evolution?** No — and the distinction matters precisely. Sprint 2's precedent (the only prior Sprint to give any Assessment-adjacent type real content) changed `Evidence`, `Finding`, and `Recommendation`'s *own* constructors while leaving `Assessment::new`'s signature completely untouched, because all three are added to the Assessment *after* construction, through their own dedicated `add_*` methods. Decision 1 (if resolved toward Option A, as recommended) is a different kind of change: the first-ever modification to `Assessment::new`'s own constructor parameters since before Sprint 1. This is not "an entity gaining content" in Sprint 2's sense; it is a change to the aggregate root's own construction contract — squarely inside `ADR-0003`'s own subject matter, not a routine content addition.

**Should `Assessment` evolve its constructor directly, versus another mechanism?** `ADR-0007`'s Runtime Entity Design Pattern establishes, for every Runtime entity so far, a single deterministic fallible constructor as the sole state-establishing mechanism — no builder pattern, no post-construction setter for structural fields. `Assessment::new` itself already follows this same shape (subject and context are supplied once, at construction, and never altered afterward — no `set_subject`/`set_context` exists). Introducing a builder or an optional setter for `VersionProfile` specifically would depart from this uniform, already-established convention for a single field, without a forcing function distinguishing it from every other constructor-supplied value. **A third constructor parameter, following the existing shape exactly, is the more consistent option** — evaluated against the established pattern, not merely the simpler one.

**Should initialization responsibility move elsewhere?** A distinct question from the constructor's own shape: should `AssessmentService` manufacture a default `VersionProfile` internally, or should it remain a caller-supplied value, mirroring how `AssessmentSubject`/`AssessmentContext` are already supplied by the CLI/Sandbox today, never manufactured inside `AssessmentService`? `GOVERNANCE.md`'s Engine boundary (*"Must never: own runtime state"*) extends naturally to *deciding which* Version Profile governs an Assessment being a business judgment, not a pure orchestration act. The existing precedent — every other constructor input is caller-supplied — favors keeping `VersionProfile` caller-supplied as well, with the application layer (CLI, Sandbox) responsible for constructing the single, minimum-viable FS25 profile named in the Capability Plan, exactly as it already constructs `AssessmentSubject`/`AssessmentContext`.

**Normal domain evolution, or an architectural boundary change?** **An architectural boundary change.** It is the first modification to `Assessment::new`'s stable signature in the platform's history, it propagates as a breaking change through both of `AssessmentService`'s public entry points (each currently accepts `subject`/`context` directly and constructs `Assessment` internally), and it directly reopens GOV-008 territory — the exact class of consequence `ADR-0009` already recorded as anticipated-but-undecided for `AssessmentService`'s entry points specifically. The correct process precedent, by direct analogy, is `ADR-0009`/GOV-008's own model: record the anticipated change and its consequences (this document does that, in evaluative form) without deciding the exact signature shape here, and require explicit governance approval — under `GOVERNANCE.md`'s Public API Policy, since `AssessmentService`'s entry points are treated as contracts — before implementation proceeds.

## Recommendation

`Assessment::new` should evolve its constructor directly (a third parameter, matching its own established shape), with the `VersionProfile` value remaining caller-supplied rather than internally manufactured by `AssessmentService`. This should be governed as an architectural boundary change, not implemented as routine domain evolution — see Architectural Decision 6 for the specific governance mechanism recommended.

## Implementation consequences

Every existing `Assessment::new` call site changes. Both `AssessmentService::execute` and `execute_from_assessment_input` require a new parameter to receive and pass through a `VersionProfile`, an unavoidable, visible signature change to both public entry points — whether this becomes the forcing function that finally resolves GOV-008, or whether a third, additive entry point is introduced instead (mirroring how `execute_from_assessment_input` itself was added alongside `execute` without replacing it), is not decided here — see Chief Architect Question 4.

# 6. Architectural Decision 5 — Crate Architecture

**Restated question:** What dependency relationships should exist for `modiq-versioning`?

## Evaluation, edge by edge

**`modiq-versioning` → `modiq-common`.** `CrateRoadmap.md`'s dependency diagram draws this edge, but `modiq-common` is, verified directly, empty and consumed by nothing in the workspace — its own role remains explicitly Open (GOV-003) for exactly that reason. Adding this edge now, with no concrete shared type to actually consume, would introduce a dependency purely for diagram-conformance, and could be read as quietly resolving GOV-003 (deciding `modiq-common` *is* the shared-types home) as a side effect of an unrelated capability, rather than through GOV-003's own governance process. **Not recommended** unless a concrete need is found and GOV-003 is separately, explicitly resolved first.

**`modiq-versioning` → `modiq-runtime`.** `VersionProfile.md`'s Non-Responsibilities (*"Does not execute Rules. Does not perform Assessments. Does not store runtime state"*) and its self-contained Conceptual Model (`GameVersion`, `Capability`, `Compatibility`) give no evidence this is needed. **Not recommended without a demonstrated need.**

**`modiq-engine` → `modiq-versioning`.** Required, directly justified: `AssessmentService` needs to hold and pass a `VersionProfile` value, mirroring the exact "compose each subsystem's own real type directly" pattern (`ADR-0010`/GOV-004) already applied identically to `modiq-collection`, `modiq-rules`, and `modiq-report`. **Recommended.**

**`modiq-rules` → `modiq-versioning`.** Required only if Architectural Decision 3 resolves toward Option B as recommended (a Rule consuming `VersionProfile` directly). Structurally identical in kind to `modiq-collection`'s own existing sole dependency on `modiq-runtime` — not a novel *pattern*, only novel for this specific crate. **Recommended, conditional on Decision 3.**

**`modiq-cli` / `apps/sandbox` → `modiq-versioning`.** Needed if the minimum-viable single profile is constructed by the application layer, per Decision 4's caller-supplied-value conclusion — mirroring the existing precedent where both already depend on `modiq-report` directly, since `modiq-engine` does not re-export `AssessmentReport` (a gap named, not yet a Governance Register item, at Engineering Releases 0.6/0.7). **Worth flagging explicitly:** if both consumers now also need a direct `modiq-versioning` dependency for the same underlying reason (`modiq-engine` not re-exporting a type its own entry points require), that is a second, independent occurrence of the identical pattern — approaching, though not yet reaching, this project's own three-point convergent-evidence bar for treating it as a real Governance Register item rather than a tracked observation.

## Layering and isolation

With `modiq-engine` → `modiq-versioning` (and conditionally `modiq-rules` → `modiq-versioning`) added, and `modiq-versioning` → `modiq-common` deliberately *not* added, `modiq-versioning` becomes a true leaf dependency — like `modiq-runtime` — consumed downward, depending on nothing, preserving the platform's unbroken strictly-downward, zero-circular-dependency record across seven Sprints (`Architecture.md`: Dependency Rules). This deliberately diverges from `CrateRoadmap.md`'s own long-standing diagram in one respect (the `modiq-common` edge) — a correction grounded in evidence of what is actually needed, not an oversight, consistent with this project's "repository truth" discipline (`CHIEF_ARCHITECT_HANDOFF_v1.0.md`, Section 7).

## Recommendation

Add `modiq-engine` → `modiq-versioning` (unconditional). Add `modiq-rules` → `modiq-versioning` only if Decision 3 resolves toward Option B. Do not add `modiq-versioning` → `modiq-common` as part of this Sprint. Confirm whether `modiq-cli`/`apps/sandbox` need a direct edge, and separately flag the recurring `modiq-engine` re-export gap for Chief Architect attention.

# 7. Architectural Decision 6 — Governance Impact

**Restated question:** Does Sprint 8 require Governance Register work, an ADR, a documentation update, or no governance action? What timing?

## Classification against `GOVERNANCE.md`'s own Change Categories

This clearly qualifies as **Level 4 (Architectural)**: it changes ownership (Decision 1), alters a stable public/constructor contract (Decision 4), and introduces new dependency edges (Decision 5) — three of the five examples `GOVERNANCE.md` itself lists for Level 4 (*"aggregate redesign," "dependency changes," "ownership changes," "major public API redesign"*). Level 4 requires, per `GOVERNANCE.md` directly: *"ADR, governance approval, documentation release, engineering implementation, engineering release."*

## Governance Register

A new item is warranted, directly analogous to GOV-009 (Assessment Input Ownership) and GOV-012 (Rule Evaluation Model) — both opened specifically to resolve "how does a new architectural concept attach to the existing model" questions of this exact shape, before any implementation began. Recommended scope (grouping, not final — a Chief Architect scoping call): Version Profile ownership (Decision 1), the Rule-Engine-input shape of version-aware evaluation (Decision 3), and the `Assessment`/`AssessmentService` construction consequence (Decision 4), whether as one item or several related ones.

## ADR

Two candidate ADR-worthy facts, evaluated separately:

- **`ADR-0003` itself would warrant a cross-reference amendment** recording `Assessment`'s new Version Profile relationship, once Decision 1 is actually decided — following the exact amendment discipline already applied to `Architecture.md`, `EngineAPI.md`, `EvidenceCollection.md`, and `DataModel.md`. Whether this rises to a *new* ADR (a durable principle beyond this Sprint) or an amendment to the existing `ADR-0003`/`ADR-0004` is itself worth a direct Chief Architect call, mirroring how GOV-012/GOV-013 were judged not to need new ADRs of their own (`CHIEF_ARCHITECT_HANDOFF_v1.0.md`, Section 4) while GOV-004 did.
- **A new `modiq-versioning` Crate Boundary Rule** — this is arguably not itself a new *decision* (`ADR-0004` already accepted Version Profiles' architectural role in full) but a documentation gap reconciling `GOVERNANCE.md` with an already-accepted decision, closer to Level 2/3 than a fresh Level 4 act. Recommended treatment: draft and review during Sprint 8's own Phase 1, without necessarily requiring a new ADR to do so.

## Documentation update

Once Decisions 1 and 3 are actually decided: `DataModel.md` and `RuleEngine.md`'s own textual tensions (Repository Evidence, above, and `SPRINT8_CAPABILITY_AND_IMPLEMENTATION_PLAN.md`'s Repository Evidence) should be resolved by an explicit, recorded amendment — not silently reconciled — per this project's "frozen does not mean immutable; amendment recorded, not silent" discipline. `GOVERNANCE.md` needs the new Crate Boundary Rule section. `CrateRoadmap.md`'s dependency diagram needs correcting to match Decision 5's actually-recommended edges, including the deliberate absence of the `modiq-versioning` → `modiq-common` edge the diagram currently shows.

## Timing recommendation

Per this project's own unbroken precedent (GOV-007 before Sprint 3 Phase 4; GOV-011 before Sprint 4 Phase 3; GOV-012 before Sprint 5 Phase 2 — proposal-then-implement, zero recorded exceptions), the Governance Register item(s) and any warranted ADR amendment should be resolved during Sprint 8's own first implementation phase (Design Preparation and Governance), strictly before any Rust code is written — never implement-then-document.

# 8. Cross-Cutting Assessment

**Recommended classification: Architectural Activation** — distinct from every classification the mission offers as an example, for a specific reason.

- **Not ordinary "Capability implementation"** in the sense Sprints 3, 4, 5, and 7 were: each of those built new capability entirely within already-real architectural surface (a new Collector inside an already-real `modiq-collection`; a new Rule inside an already-real `modiq-rules`), touching zero previously-stable public signatures. This capability, by contrast, is very likely the first to force a change to `Assessment::new`'s own constructor (Decision 4) and to both of `AssessmentService`'s public entry points — consequences no prior capability Sprint has required.
- **Not "Infrastructure expansion"** in the sense of building new generic machinery for its own sake — this project's "capability before abstraction" discipline explicitly rejects that motivation (Decisions 2, 3, and 5 each reject an alternative precisely because it would have introduced unjustified new machinery: a second Collector, a dedicated Compatibility Service, a speculative `modiq-common` dependency).
- **Narrower than "Platform evolution,"** which implies broader, more open-ended architectural change than this scoped, single capability actually requires — though it shares real DNA with platform evolution in one respect: it is plausibly the forcing function that finally resolves GOV-008, a genuinely platform-level consequence, five Sprints in the making.
- **"Architectural Activation" names the actual shape of the work precisely:** `modiq-versioning` and the Version Profile concept have been fully specified since Documentation Release 1.0 (`VersionProfile.md`, `ADR-0004`) and named as a dependency of `AssessmentService` since `EngineAPI.md`'s own amendment (*"Version Profiles (once implemented)"*) — but never built. This Sprint is the first to realize an already-specified-but-dormant architectural dimension, not to invent a new one or merely extend an already-live one.
