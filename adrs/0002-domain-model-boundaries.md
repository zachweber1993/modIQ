# ADR-0001: Runtime Domain vs Knowledge Domain

| Property | Value |
|----------|-------|
| **ADR** | 0001 |
| **Title** | Separate Runtime Domain from Knowledge Domain |
| **Status** | Accepted |
| **Project** | modIQ |
| **Date** | 2026-07-15 |

---

# Context

modIQ represents two fundamentally different kinds of information:

1. Information generated by an Assessment.
2. Engineering knowledge that exists independently of any Assessment.

Without an explicit boundary, future specifications risk mixing runtime entities with knowledge entities, increasing coupling and reducing architectural clarity.

---

# Decision

The platform shall distinguish between:

## Runtime Domain

Entities that exist because an Assessment is executed.

Examples include:

- Assessment
- Assessment Subject
- Assessment Context
- Evidence
- Finding
- Recommendation
- Assessment Report
- Version Profile

These entities are defined within DataModel.md.

---

## Knowledge Domain

Entities that represent reusable engineering knowledge independent of any Assessment.

Examples include:

- Rule
- Repair Recipe
- Best Practice
- Engine Behavior
- Compatibility Pattern
- Knowledge Reference

These entities are defined within KnowledgeModel.md.

---

# Consequences

Benefits include:

- Clear separation of responsibilities.
- Reduced coupling.
- Simpler Rule Engine design.
- Improved extensibility.
- Cleaner domain model.
- Independent evolution of runtime data and engineering knowledge.

---

# Status

Accepted.

This decision is authoritative for Documentation Release 1.0.
